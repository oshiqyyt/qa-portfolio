# Баг-репортинг

Пример качественного баг-репорта, оформленного по профессиональному стандарту.

## Задание (task_1.md)

**Объект:** [saucedemo.com](https://www.saucedemo.com/)

**Что сделано:**

### 1. Поиск дефекта
- Проведено исследовательское тестирование страницы авторизации
- Найден и воспроизведён дефект
- Проверено на уникальность (отсутствие дублей)

### 2. Оформление баг-репорта
**Структура:**
- **Заголовок** по принципу «Что? Где? Когда?»
- **Серьёзность** (Severity) — влияние на бизнес-логику
- **Приоритет** (Priority) — срочность исправления
- **Шаги воспроизведения** — подробное описание действий
- **Ожидаемый результат** — как должно работать
- **Фактический результат** — что происходит на самом деле
- **Вложения** — скриншоты, логи

### 3. Анализ дефекта
- Обоснование, почему это баг, а не фича
- Гипотеза о возможных причинах возникновения
- Оценка рисков для пользователя и бизнеса
- Точки исправления (техническое размышление)

## Применённые навыки

- Локализация дефектов
- Чёткое и структурированное описание проблемы
- Приоритизация багов (Severity vs Priority)
- Аргументация важности исправления
- Работа с вложениями (скриншоты, доказательства)
